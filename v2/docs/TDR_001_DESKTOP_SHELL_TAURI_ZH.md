# TDR-001：桌面壳选型决策（Tauri）

状态：Accepted  
日期：2026-02-24  
关联文档：`docs/DESKTOP_OPEN_SOURCE_BENCHMARK_ZH.md`、`docs/DESKTOP_MIGRATION_PLAN_ZH.md`

## 1. 背景

当前 V2 采用本地开发式启动（Designer/API/Agent 分别启动），不符合非开发用户“一键可用”预期。  
需要新增桌面壳统一托管本地服务，并保持现有核心架构不重写。

## 2. 决策

采用 `Tauri` 作为桌面壳主线方案，首发 Windows。

## 3. 决策依据

1. 能直接承载现有 React Designer，迁移成本低。
2. 能满足本地服务托管、健康检查、日志与目录管理需求。
3. 包体与资源占用相对可控，符合本地优先工具定位。
4. 许可证友好（MIT/Apache-2.0），可用于产品化交付。

## 4. 影响范围

新增：

1. `apps/desktop`（桌面壳项目）。
2. 壳层进程编排（API/Agent 启停、健康检查、重启策略）。
3. 本地数据目录与诊断包机制。

调整：

1. Designer 读取运行时 `API_BASE`，取消硬编码依赖。
2. API 增强健康检查元信息（版本/状态/依赖）。
3. Agent 增强日志规范与退出信号处理。

不变：

1. `Designer + API + Agent + Recorder` 业务分层。
2. 录制扩展主流程与 selector 策略。

## 5. 风险与缓解

风险 1：Rust 工具链增加上手成本。  
缓解：提供标准化本地 bootstrap 脚本和 CI 构建说明。

风险 2：壳层进程托管不稳定。  
缓解：先实现最小托管，再引入健康检查与退避重试。

风险 3：桌面壳与扩展地址配置不一致。  
缓解：统一本地地址配置中心，增加连接自检与中文提示。

## 6. 回滚与替代

1. 若阶段 1/2 触发阻断条件，评审切换到 `Electron`。
2. 始终保留当前 CLI 启动链路作为 fallback。
3. 按阶段标签回滚，不做跨阶段混合回退。

## 7. 验收标准

1. 一键启动后可在 10 分钟内完成“创建/校验/运行/查看日志”闭环。
2. 8 小时冒烟运行无静默失败。
3. 用户不需要手动开多个终端，不需要手写 selector。
