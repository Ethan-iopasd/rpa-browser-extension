# 桌面化开源对标清单（阶段 0 强制）

更新时间：2026-02-24

## 目标问题

在保持现有 `Designer + API + Agent + Recorder` 架构不重写的前提下，实现桌面端一键启动与本地服务托管，满足：

1. 非开发用户安装后可直接使用。
2. 可稳定托管 Python 本地进程（API/Agent）。
3. 打包发布与升级路径可落地。
4. 与现有前端技术栈兼容，改造成本可控。

## 候选方案与对标结论

### 1) Tauri（首选）

- 许可证：
  - MIT / Apache-2.0。
- 可复用模块：
  - 窗口与 WebView 容器。
  - 系统路径（用户数据目录）管理能力。
  - 命令调用与前后端桥接机制（用于进程编排与健康检查）。
  - 构建与打包链路（Windows 安装包）。
- 与本项目匹配度：
  - 高：可直接承载现有 React Designer。
  - 高：可通过壳层管理 API/Agent 生命周期。
  - 高：包体较轻，适合本地优先工具。
- 接入风险：
  - 需要引入 Rust 工具链，团队需补齐构建环境。
  - 进程托管与跨平台细节需要早期验证。
- 结论：
  - 作为主线方案推进。

### 2) Electron（备选）

- 许可证：
  - MIT。
- 可复用模块：
  - 成熟的桌面壳能力与生态插件。
  - 主进程管理子进程、自动更新、崩溃上报能力。
- 与本项目匹配度：
  - 高：接入 React 成本低，开发体验成熟。
  - 中：功能足够，但对当前目标存在“能力过剩”。
- 接入风险：
  - 安装包与运行时体积较大。
  - 资源占用更高，影响轻量本地工具体验。
- 结论：
  - 可作为风险兜底，不作为首选。

### 3) Wails（备选）

- 许可证：
  - MIT。
- 可复用模块：
  - Go + Web 前端桌面壳模式。
  - 基础窗口与打包能力。
- 与本项目匹配度：
  - 中：需要额外引入 Go 技术栈，与现有 Python/TS 主栈不一致。
  - 中：生态与社区体量小于 Tauri/Electron。
- 接入风险：
  - 团队工具链与维护复杂度上升。
  - 二次开发与问题检索成本较高。
- 结论：
  - 不作为当前阶段主线。

### 4) Neutralinojs（反例）

- 许可证：
  - MIT。
- 可复用模块：
  - 轻量桌面容器与基础 API。
- 与本项目匹配度：
  - 中偏低：轻量优势明显，但进程编排与工程化能力相对弱。
- 接入风险：
  - 在复杂托管场景下可扩展性不足。
  - 发布和运维生态不如主流方案成熟。
- 结论：
  - 不进入主线，仅保留概念参考。

## 统一取舍标准（用于决策）

1. 是否最小改造复用现有代码（优先）。
2. 是否稳定托管本地服务并支持诊断（必须）。
3. 是否支持 Windows 安装包快速发布（必须）。
4. 是否控制包体与资源占用（重要）。
5. 是否降低长期维护复杂度（重要）。

## 决策结果

1. 主线采用：`Tauri`。
2. 兜底备选：`Electron`（仅当 Tauri 阶段 1 出现阻断性问题时切换）。
3. 本阶段不采用：`Wails`、`Neutralinojs`。

## 切换触发条件（从 Tauri 切到 Electron）

满足任一条件即触发评审：

1. 阶段 1 无法稳定完成 API/Agent 进程托管。
2. 阶段 2 无法满足 8 小时冒烟稳定性目标。
3. 出现无法规避的打包/签名阻断，影响 Beta 交付节奏。

## 阶段 1 前置清单

1. 固化 TDR 文档并评审通过。
2. 建立 `apps/desktop` 骨架分支。
3. 明确数据目录、日志目录、诊断包目录规范。
