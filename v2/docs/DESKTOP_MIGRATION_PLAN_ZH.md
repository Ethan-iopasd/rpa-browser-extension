# 桌面化改造计划（遵循产品第一原则）

更新时间：2026-02-24

## 1. 目标与范围

目标：在不重写现有核心能力的前提下，将 V2 改造为“桌面壳 + 本地服务”的一键可用形态，降低非开发用户上手门槛。

范围内：

1. 新增桌面壳应用（首发 Windows）。
2. 自动启动并托管 `services/api` 与 `apps/agent`。
3. `apps/designer` 作为桌面壳内主界面运行。
4. 保留 `apps/recorder-extension` 录制链路。
5. 建立安装、升级、诊断、日志导出基础能力。

范围外（本阶段不做）：

1. 重写 UI 为纯原生桌面（PySide/Qt 等）。
2. 改造为云端多租户部署。
3. 移除浏览器扩展后重做录制能力。

## 2. 强约束（必须满足）

1. 非开发用户优先：安装后可一键启动，不需要手动开多个终端。
2. 开源优先复用：先完成桌面壳方案对标调研，再进入编码。
3. 不要求手写代码：核心流程保持可视化完成。
4. 复杂页面能力不回退：iframe、弹窗、新标签页链路保持可用。
5. 核心中文化：主要按钮、提示、错误信息使用中文并给出下一步建议。

## 3. 技术路线

推荐路线：`Tauri + 现有 React/FastAPI/Agent`（壳化集成，不重写内核）。

架构关系：

1. Desktop Shell：进程编排、健康检查、日志、配置、升级。
2. Designer：继续负责编排与控制台交互。
3. API：流程校验、运行管理、任务管理、安全管理。
4. Agent：执行 Playwright 流程并回传事件。
5. Recorder Extension：继续负责页面录制与 selector 采集。

## 4. 分阶段实施计划

### 阶段 0：开源对标与决策（1 天）

任务：

1. 对标至少 3 个方案：Tauri、Electron、Neutralinojs/Wails。
2. 记录许可证、可复用模块、接入风险、放弃理由。
3. 形成决策文档并冻结首选方案（建议 Tauri）。

交付：

1. `docs/DESKTOP_OPEN_SOURCE_BENCHMARK_ZH.md`
2. 技术决策记录（TDR/ADR）一份。

验收：

1. 未完成对标文档不得进入阶段 1 编码。

### 阶段 1：最小可用桌面壳（3-4 天）

任务：

1. 新建 `v2/apps/desktop`（Tauri 项目骨架）。
2. 壳启动时自动拉起 API 进程，并加载 Designer。
3. Designer 通过配置读取 `API_BASE`（避免硬编码）。
4. 提供最小状态栏：服务启动中/已就绪/失败。

交付：

1. 一键启动可进入 `/dashboard`。
2. 可完成“流程校验 + 启动运行 + 查看运行详情”基础闭环。

验收：

1. 新用户在 10 分钟内完成首次运行闭环。

### 阶段 2：运行托管与数据目录（3-4 天）

任务：

1. 壳层增加 API/Agent 健康检查与异常重启。
2. 统一用户数据目录（配置、流程、运行日志、诊断包）。
3. 增加端口占用、权限异常、服务未就绪的中文诊断提示。
4. 新增“导出诊断包”能力用于支持排障。

交付：

1. 本地服务稳定托管，不依赖手动命令行守护。
2. 数据不再依赖仓库目录，用户环境可独立运行。

验收：

1. 连续 8 小时冒烟运行无静默失败。
2. 常见错误可在 UI 内定位并给出操作建议。

### 阶段 3：安装、升级与发布（2-3 天）

任务：

1. 打包安装程序（Windows）。
2. 增加版本信息页与更新提示。
3. 固化发布脚本与发布检查清单。

交付：

1. 内部 Beta 安装包。
2. 发布文档与回滚步骤文档。

验收：

1. 干净机器可完成安装、启动、运行、卸载全流程。

## 5. 工作拆分（按模块）

`apps/desktop`：

1. 进程生命周期管理（start/stop/restart）。
2. 健康检查与重试策略。
3. 错误分类与中文提示。
4. 日志聚合与诊断导出。

`apps/designer`：

1. `API_BASE` 与运行时配置注入。
2. 服务状态提示组件。
3. 诊断入口与帮助入口。

`services/api`：

1. 健康检查端点增强（版本、依赖状态、启动耗时）。
2. 错误码与错误信息规范化。

`apps/agent`：

1. 启动/关闭信号处理。
2. 运行时日志结构统一（便于桌面壳收集）。

`apps/recorder-extension`：

1. 与桌面模式兼容的本地地址配置策略。
2. 连接失败时中文引导提示。

## 6. 质量门禁

每个阶段至少满足：

1. `pnpm lint`
2. `pnpm typecheck`
3. `pnpm test`
4. `pnpm build`
5. 新增桌面链路冒烟：启动桌面 -> 校验流程 -> 发起运行 -> 查看日志

补充门禁：

1. 不允许引入要求用户手写 selector 的回退方案。
2. 不允许破坏现有 iframe/弹窗/新标签页录制与回放能力。

## 7. 风险与应对

风险 1：桌面壳进程管理导致启动不稳定。  
应对：增加健康检查、超时重试、可视化诊断入口。

风险 2：本地路径和权限差异导致数据读写失败。  
应对：统一数据目录并在首启做可写性检测。

风险 3：录制扩展与桌面部署地址不一致。  
应对：扩展侧增加动态地址配置和连接自检。

风险 4：改造期间回归风险高。  
应对：保留当前 CLI 启动链路作为 fallback，分阶段灰度。

## 8. 回滚策略

1. 保留现有 `v2` 开发启动方式（API + Designer + Agent 手动启动）。
2. 每阶段独立分支提交，按阶段可逆回退。
3. 任何阶段若核心闭环失败，立即回退到上一个可用标签继续修复。

## 9. 建议分支策略

1. `feat/desktop-shell-bootstrap`（阶段 1）
2. `feat/desktop-runtime-supervisor`（阶段 2）
3. `feat/desktop-packaging-release`（阶段 3）
4. `release/beta-desktop`（发布整合）

## 10. 里程碑检查点

1. M1：桌面壳可启动并连通 API（阶段 1 完成）
2. M2：桌面壳稳定托管 API/Agent，8 小时冒烟通过（阶段 2 完成）
3. M3：安装包可在干净机器完整跑通（阶段 3 完成）
